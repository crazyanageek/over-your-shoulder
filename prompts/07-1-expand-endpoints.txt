Phase 07.1 — Expand endpoint coverage for AI services and third-party monitoring

=== GOAL ===

Expand OYS from 27 observed domains to ~80-90, covering:
- Additional AI services (use.ai, Character.AI, Poe, Cursor, Codeium, ElevenLabs, Runway, etc.)
- Third-party monitoring/tracking services that many AI platforms embed (Datadog, Sentry, Segment, Statsig, etc.)

Introduce an internal "type" field on each endpoint entry, for future product exploitation. Zero editorial changes to the PDF report or badge for this phase — new events just increase the counts in existing buckets.

=== SCOPE DECISIONS (agreed with Jon) ===

- ~50 AI services covered (adding ~23 to the current 27)
- ~25 monitoring/tracking services covered (adding a new category)
- Typology introduced in data but NOT surfaced in UI/PDF for this phase
- Minimal changes outside of data/endpoints.json, manifest.json, and the service worker's categorization logic
- Report text, badge, scoring formula, landing page all UNCHANGED

=== STEP 2 — ADD "type" FIELD TO EACH ENTRY ===

Add a new string field `type` to every entry. Values:
- "ai_direct" — service that directly processes user prompts (ChatGPT, Claude, Gemini, Mistral, etc.)
- "monitoring" — RUM / error tracking / session replay (Datadog, Sentry, LogRocket, FullStory, Hotjar, Clarity)
- "analytics" — user behavior analytics (Segment, Mixpanel, Amplitude, PostHog, Heap, GA, GTM, Plausible, Fathom)
- "feature_flags" — experimentation/feature flag services (Statsig, LaunchDarkly)

Backfill all 27 existing entries with `type: "ai_direct"`.

The `category` field keeps its current values ("ai_chat", "ai_api", etc.) which are about the user-facing nature of the service. The new `type` field is about the architectural role.

=== STEP 6 — UPDATE service worker's categorization ===

Open background/service_worker.js. Find the webRequest listener or wherever endpoints are matched to metadata.

When an event is captured, the counter should increment in whatever buckets exist today (byVendor, byCountry, byHost). THAT STAYS THE SAME.

The only addition: also store the `type` field of the matched endpoint in the event record. So each oys_events_YYYYMMDD entry now looks like:

  {
    ts: 1728123456789,
    hostname: "browser-intake-us5-datadoghq.com",
    pathname: "/api/v2/rum",
    tabId: 42,
    initiator: "https://claude.ai",
    vendor: "Datadog",          // new — or derived from endpoints.json
    country: "US",               // already there
    category: "monitoring_rum",  // already there
    type: "monitoring"           // NEW — this field
  }

Adapt the byVendor counter: currently it buckets by vendor. For monitoring/analytics services, vendor is the service itself (Datadog, Sentry, Mixpanel, etc.).

IMPORTANT: when a monitoring event is captured from claude.ai (initiator: claude.ai), we should NOT double-count. Each observed outbound request is one event, period. The initiator is just context.

=== STEP 7 — UPDATE lib/score.js if needed ===

The scoring formula (4 factors: Volume, Categories, Geography, Continuity) doesn't change.

But verify: does `categoriesScore` currently count distinct categories from counters.byCategory? If yes, it will now see new categories (monitoring_rum, analytics_events, etc.) and could give a higher categoriesScore than before for the same user.

TWO OPTIONS:

Option A — Don't let monitoring/analytics types contribute to categoriesScore.
Only ai_direct events contribute to the "sensitive categories touched" count.

Option B — Let them contribute. User sees their categoriesScore increase because now OYS captures more categories.

Decision needed from Jon. For this phase, go with Option A (minimal surprise to existing users, preserves current scoring semantics). Implement: filter counters.byCategory to only include types matching ai_direct when computing categoriesScore.
